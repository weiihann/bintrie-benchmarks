Section 1

Executive Summary

PBT outperforms UBT on every benchmark, and the gap is large. 1.75× Mgas/s on reads (p=1.5e-147), 2.83× on writes (p=1.6e-128), 2.41× on mixed (p≈0). Every Mann-Whitney test is significant; bootstrap 95% CIs for throughput exclude 1.0 by a wide margin. The same-state guarantee holds (aggregate gas-per-slot ratio = 1.0000) — the EVM did identical work in both configs; only the trie key layout changed.

Both configs run the same binary trie (EIP-7864) at group depth 5 with bitarray path encoding, and both now serve state reads from a flat-state snapshot (a 'F'-prefixed key/value store mapping a 31-byte stem to its packed values) rather than walking the trie. The only variable under test is what happens between an EVM storage key and its on-disk key:

  • UBT (Unified Binary Trie) — one key-derivation function for everything; a contract's storage slots are scattered across the whole keyspace by a full hash.
  • PBT (Partitioned Binary Trie) — account / code-chunk / storage-slot each derive into their own zone, so a contract's storage slots share a key prefix and land contiguously on disk.

The result flips the original hypothesis. We expected flat-state to shrink PBT's read advantage: if reads no longer traverse the trie, PBT's shorter zone paths shouldn't matter. Instead the read lead grew — from +11% in the earlier trie-traversal study (group depth 8) to 1.75× here. Section 7 shows why: the same zone partitioning that shortened trie paths also clusters each contract's flat-state blobs on disk, so the advantage simply migrated from trie depth to disk locality.

Section 2

The Two Designs

Both UBT and PBT use the same on-disk trie structure (binary trie, group depth 5, bitarray path encoding) and the same flat-state read path. A storage read first checks the flat-state snapshot — key 'F' || stem(31), value bitmap(32) || packed values — and only falls back to the trie on a miss. So for reads, the trie shape is irrelevant; what matters is where the flat-state blob for a given slot physically lands in Pebble. That is determined entirely by how the slot's stem is derived.

UBT — Unified key derivation

A storage slot's key is sha256(0×12 || addr || (MAIN_STORAGE_OFFSET + slot)). The slot number is hashed in, so two consecutive slots of the same contract produce keys that are essentially random with respect to each other. A contract's storage is therefore scattered uniformly across the 256-bit keyspace — and across the whole Pebble SSTable layout on disk.

PBT — Zone-partitioned key derivation

The slot key is built, not hashed whole. The top bits carry a zone tag and the top 60 bits of H(addr); the slot's own hash fills the middle, the sub-index the bottom:

key bits (PBT storage zone): [255] zone = 1 (storage) [254..195] top 60 bits of H(addr) ← same for ALL slots of a contract [194..8] top 187 bits of H(addr||tree_index) [7..0] slot % 256

Because the high-order bits are identical for every storage slot of a given contract, all of that contract's slots sit contiguously in Pebble's sorted keyspace. The same mechanism puts account fields and code chunks in their own zones. (PBT also commits the three zone subtries in parallel, which helps the write path — but as Section 7 shows, the dominant effect here is the read-side locality.)

The weighted multi-contract workload

Rather than hammer a single contract, the benchmark deploys 10 ERC20 contracts and visits them on a deterministic power-law schedule: a few "hot" contracts are benchmarked many times, a long tail once — mirroring real-world contract popularity. The schedule (a seeded Zipf weighting, interleaved so the hot contract isn't fired back-to-back) is a pure function of the seed and contract count, so it is byte-identical for UBT and PBT. Each visit is one cold-cache geth session against one contract.

The same-state guarantee

UBT and PBT share the state-actor seed and the spamoor seed, so the EVM-level workload is byte-identical: same accounts, same contracts, same storage layout, same 10 ERC20s with the same holders (the deployed contract addresses are asserted identical across configs). The two configs only differ in how a storage access maps to an on-disk key.

Correctness anchor: the aggregate gas-per-slot (total gas ÷ total storage-slot accesses) is 1.0000 for all three benchmarks, and total slots accessed match to within 0.003%. Same logical work per slot in both configs — the timing comparison measures the key layout, not the workload.

Caveat: per-block gas_used sequences are not byte-identical, because the dev chain seals a block every second and a faster config packs more work per block. This block-boundary jitter also makes the per-block gas_per_slot ratio for mixed read 1.46 — an artifact of two regimes (cheap SLOADs, expensive SSTOREs) partitioning into blocks differently. The aggregate (1.0000) is the correct same-state metric and confirms the workloads are identical.

Section 4

Reads — erc20_balanceof

The benchmark calls balanceOf() in a loop against incrementing addresses, until ~100M gas of work is done. Each call is a cold flat-state read of a different storage slot. No trie traversal, no writes — pure read latency against the 'F'-prefixed snapshot in a ~73 GB Pebble store.

State reads dominate, and that is exactly where PBT wins: read time drops 61%. The tiny rises in hash/commit are per-block bookkeeping (three zone roots cost a hair more than one) and are negligible next to the read saving.

No block-size artifact on reads. Both configs produce identical blocks here — 16.7M gas, 1 transaction each — and within each config throughput is uncorrelated with block size (r≈0.0). So the 1.75× is not a measurement bias from PBT packing bigger blocks; it is a genuine per-read difference.

Per-slot read cost: UBT 60.1 µs/slot vs PBT 21.9 µs/slot = 2.7× faster (ratio 0.365, CI [0.356, 0.372], p=2.2e-157). Per cache-miss (i.e. per actual disk read): 2.4× faster (ratio 0.414).

Section 5

Writes — erc20_approve

The benchmark calls approve(spender, value) in a loop. Each call is a cold read of the current allowance followed by a cold-to-warm SSTORE. A bump pointer in storage advances every run, so each run writes a fresh chunk of slots — never overwriting a prior run's (gas-per-slot holds constant across runs).

UBT: 31.9 Mgas/s — PBT: 90.2 Mgas/s — 2.83× (Mann-Whitney p=1.6e-128, CI [2.775, 2.861])

Writes show the largest gap. Both PBT advantages stack here: the read half of each approve benefits from flat-state locality, and the write half benefits from parallel zone commit.

Every per-slot component is dramatically lower for PBT. Reads are 4.4× cheaper per slot (locality); hashing is 2.4× cheaper per slot (parallel zone commit). These per-slot metrics are normalized by slots touched, so they are immune to the block-packing difference noted below.

Block-size asymmetry is present here — but is not the driver. Because PBT executes faster, its 1-second dev blocks pack more work (PBT ~50M gas / 3 tx per block vs UBT ~25M / 2 tx). Throughput is only weakly correlated with block size within a config, and the per-slot metrics above — which divide out block size entirely — still show 2.4–4.4× PBT advantages. The packing difference is a consequence of PBT being faster, not a cause of the measured gap.

The read win is disk locality

Decomposing the read-time gap (block-size-independent): PBT is ~2.4× faster per cold disk read, and additionally takes ~1.13× fewer reads (higher cache hit rate). The fewer-reads piece accounts for only ~13% of the gap; the dominant factor is each disk read being cheaper. Since the flat-state blob format is byte-identical between configs, the only thing that can make a read cheaper is where the blob sits on disk.

That traces directly to the key derivation (Section 2). For a benchmark that reads many slots of one contract, PBT's keys — sharing a zone + H(addr) prefix — are contiguous in Pebble, so the reads become a near-sequential SSTable scan with strong page locality. UBT's keys — a full hash of (addr, slot) — are scattered, so the same reads are random seeks across a 73 GB store. The higher in-memory cache hit rate for PBT (22% vs 13%) is a symptom of the same clustering, not an independent effect.

Patterns that hold

Pattern 1: the advantage migrated, it didn't disappear. Pre-flat-state (trie traversal, GD8), PBT's read win was +11% from shorter zone paths. Flat-state removes trie traversal from reads entirely — yet the read win grew to 1.75×, because zone partitioning also clusters flat-state blobs on disk.

Pattern 2: parallel commit adds to it on writes. PBT's per-slot hash cost is 0.41× on approve and 0.71× on mixed — the three zone subtries hashing concurrently. This stacks on top of the read-locality win, which is why writes show the biggest overall gap (2.83×).

Pattern 3: the workload is identical. Aggregate gas-per-slot = 1.0000 on all three benchmarks; total slots accessed match to 0.003%. PBT's gain is genuine layout engineering, not a hidden workload or gas-schedule shift.

Section 8

Verdict

Zone partitioning is a large win — especially with flat state.

1.75×–2.83× Mgas/s across all three benchmarks, with significance far above the noise floor and CIs that exclude 1.0 by a wide margin. The same-state guarantee holds. The win is driven primarily by flat-state disk locality: clustering a contract's storage blobs turns random seeks into sequential reads. Parallel zone commit adds a second, smaller win on the write path.

Caveats worth keeping in mind

  • The workload favors per-contract locality — on purpose. Each invocation reads/writes many slots of one contract, which is exactly where PBT's per-contract clustering pays off. This is realistic for ERC20 traffic (many holders of one token), and the power-law schedule mirrors real contract skew — but a workload that touched one slot each across many random contracts would see a smaller PBT advantage.
  • Block-size asymmetry on writes/mixed. PBT's faster execution packs ~2× more work per 1-second dev block. The per-slot metrics divide this out and still favor PBT, but raw per-block throughput is partly amplified by it.
  • Read-benchmark variance. PBT's erc20_balanceof CV is 15% over 5 runs (cold read latency is disk-cache sensitive). The 1.75× headline holds comfortably, but with wider error bars than the write benchmark.
  • Scale. 73 GB DBs on an 8-core box with 31 GB RAM — a useful mid-scale result. The magnitude of the locality win may shift at mainnet scale and with different SSD/RAM ratios.

Open questions

  • How much of the locality win survives when the working set fits in RAM (warm cache), versus this cold-cache, seek-bound regime?
  • Does the advantage shrink for workloads that spread thinly across many contracts rather than concentrating per contract?
  • Can UBT recover most of the gap with a locality-preserving key derivation, without full zone partitioning?
