Benchmark Report — Scattered-Access Decomposition

UBT vs PBT: Locality vs Parallel Commit

A scattered-access campaign that isolates where PBT's win actually comes from. With each operation hitting a different contract, PBT's read advantage disappears (and even reverses) — that was disk locality, not trie speed. Its write advantage persists as a clean ~1.4× — that part is real parallel-zone commit. The two together decompose PBT's behaviour into a workload-dependent locality bonus + a structural commit win.

Section 1

Executive Summary

The headline decomposes PBT's edge into two distinct effects. Reads (where the trie has little real hashing to do): PBT is actually 0.84× — slower — because its 3-zone-root overhead has nothing to amortise once scattering removes the disk-locality bonus. Writes (where there's real hashing to commit): PBT is 1.43× faster — the parallel zone commit is a structural win that survives scattering. Mixed sits between them at 1.05× (not significant).

Both configs run the same binary trie (EIP-7864) at group depth 5, both serve state reads from the flat-state snapshot, and both operate on the same 75 GB DB. The one variable is the key derivation:

  • UBT (Unified Binary Trie) — one keyspace; a contract's storage slots scatter across it by a full hash.
  • PBT (Partitioned Binary Trie) — account/code/storage zones, with a zone + H(addr) high-bit prefix so a contract's storage slots cluster contiguously in Pebble; the three zone subtries commit in parallel.

Section 2

The Two Designs and the Scattered Probe

Both UBT and PBT use the same on-disk trie structure (binary trie, group depth 5, bitarray path encoding) and serve state reads from a flat-state snapshot keyed by 31-byte stem. They differ only in how a storage access becomes that stem.

UBT — unified key derivation

A slot's stem is essentially sha256(addr || slot) — the slot number is hashed in, so consecutive slots of the same contract land at scattered places in the keyspace, and across the Pebble SSTable layout on disk.

PBT — zone-partitioned key derivation

The slot's stem is built, not hashed whole. The top bits carry a zone tag and the top bits of H(addr); the slot's own hash fills the middle:

key bits (PBT storage zone): [255] zone = 1 (storage) [254..195] top 60 bits of H(addr) ← same for ALL slots of a contract [194..8] top 187 bits of H(addr||tree_index) [7..0] slot % 256

Because the high bits are identical for every slot of a given contract, all of its storage sits contiguously in Pebble's sorted keyspace. Account fields and code chunks each have their own zones. PBT also commits the three zone subtries in parallel.

The scattered probe (this campaign)

To isolate where PBT's advantage comes from, the benchmark forces each EVM operation to land in a different contract. Concretely:

  • Each invocation is one transaction that walks an in-memory table of 2000 distinct contract addresses and CALLs a different one every iteration.
  • The 2000 target contracts are tiny getter contracts (CREATE2-deployed, identical bytecode, addresses recomputed deterministically in Python). On call they execute SLOAD or SSTORE against their own storage based on a 96-byte calldata [slot, value, isWrite].
  • Each block touches 2000 distinct contracts — so PBT cannot stream a single contract's clustered storage; every blob fetch lands in a different cluster.

Why getters and not real ERC20s: spamoor's erc20_bloater takes ~2.5 min per contract (200-wallet distribution), so 2000 ERC20s would take ~9 days. spamoor's factorydeploytx deploys 2000 CREATE2 getters in ~20 min. The getter does raw SLOAD/SSTORE, not ERC20 balanceOf/approve — it's a clustering probe, not a faithful ERC20 workload.

The two methodology fixes (mirroring mpt-vs-bintrie)

Fix 1 — one transaction per block. With --dev.period 1, a faster config packs more transactions into each 1-second block; per-block fixed costs (state-root commit) amortise over more gas and inflate Mgas/s. We size --gas-benchmark-values 16 so each invocation is exactly one tx (~16M gas, Osaka's per-tx cap), and use --dev.period 10 to give that tx its own block. Result: every block is byte-identical structure across configs (validated 15,900,160 gas balanceof / 15,907,879 approve), and Mgas/s is apples-to-apples. This is exactly the structure mpt-vs-bintrie called "the cleanest comparison in the study."

Fix 2 — cold-insert writes via a per-run counter offset. The attack contract's slot counter used to live in its own storage, but it's redeployed each invocation — so the counter reset and runs 2+ re-wrote the same slots (warm updates). The harness now passes SCATTERED_WRITE_OFFSET = run × 100M as the first calldata word; the attack starts its counter there. Each run writes a never-used slot range — cold inserts (~28,400 gas/slot, vs ~5k for warm updates). Identical across configs, so the same-state guarantee holds.

The same-state guarantee

state-actor seed 25519, deterministic CREATE2 getters (identical factory + initcode hash), and the same per-run offset both configs — so the on-chain workload is byte-identical. Asserted in two places:

  • contracts.json is byte-identical across configs (2000 addresses each, same factory 0x7dc4d193…, same initcode hash 0x56ee07c5…).
  • Aggregate gas_per_slot ratio = 1.0000 for all three benchmarks (vs PBT/UBT throughput which differs because the trie does).

Run-to-run stability (CV%)

Storage cache hit-rates are equal between configs (24.5 vs 23.5% reads, 31 vs 30% writes) — the cache-asymmetry artefact that troubled the earlier concentrated run is absent here, a side-effect of the 1-tx-per-block fix.

Reads — scattered_sload

The attack contract CALLs a different one of 2000 getters every iteration, each call doing a single SLOAD(slot 0) against that getter's storage. With each block touching 2000 distinct contracts, PBT cannot stream any single cluster — every flat-state blob fetch is a fresh random location in the 72 GB DB.

The read cost itself is equal (31 ms = 31 ms). Per-cache-miss ratio is 0.97, per-slot-read 0.97 — PBT is effectively indistinguishable from UBT on the actual flat-state fetch. The locality advantage that gave PBT a 1.75× read win on concentrated access is fully gone.

What's left is the per-block overhead PBT pays for being a 3-zone trie: even on a read-only block (no real hashing to do), state-root commit must touch all three zone subtries (15 ms hash + 6 ms commit) instead of UBT's single root (2 ms + 1 ms). On these small, read-only blocks, that overhead is most of the difference, and it puts PBT 20% behind in total wall-time.

UBT pulls ahead on reads under scattered access. Each box is 30 cold 1-tx blocks; box widths are tight because every block has byte-identical structure.

Per cache miss the two configs are indistinguishable — consistent with locality being the only thing PBT used to win on reads.

Section 5

Writes — scattered_sstore

Same sweep, but each operation writes a fresh slot (cold insert; the harness's per-run offset guarantees the slot has never been written before). Each block scatters ~560 cold inserts across the 2000 getters.

Now there's real work for the trie to do (an insertion path per write, dirty nodes to hash, three zones to commit), and PBT's parallel-zone commit pulls clear. The 1.43× throughput maps onto a clean 30% per-block speedup. Per-slot ratios match: ms_per_slot_hash = 0.82, ms_per_slot_total = 0.21 — structural and block-size-independent.

UBT's higher state_read_ms on writes (116 vs 31) is the read-before-write that SSTORE requires; the insertion-path traversal benefits from PBT's clustering even though we're scattering across contracts, because each individual insert's path lookup happens within one contract's cluster.

Section 6

Mixed — scattered_mixed

Two phases per invocation (read sweep then write sweep) across 5 gas ratios. The result lands between reads (0.84×) and writes (1.43×), where a gas-weighted average would put it.

UBT: 106.4 Mgas/s — PBT: 111.4 Mgas/s — 1.05× (Mann-Whitney p=0.23, not significant)

The PBT/UBT ratio is small enough that with 30 blocks per cell the test cannot rule out chance — the per-cache-miss ratio (0.55) and per-slot-read (0.56) suggest PBT does have an edge on the read-half of mixed (where the insertion-path locality helps), but the wins and losses largely cancel at the block level.

Section 7

Why — fixed overhead vs parallelism

The 1-tx-per-block fix didn't just clean the numbers, it revealed the actual mechanism, which block-packing had been hiding. PBT's three-zone trie design imposes a higher fixed per-block cost (three state roots instead of one) but it parallelises the variable hashing work. Whether that's a net win depends on how much hashing the block actually has to do.

Reads: little to hash. PBT's 3-zone root overhead dominates the difference → PBT loses 0.84×. The famous concentrated-access read win (1.75×) was entirely disk locality — remove it and PBT is structurally behind.

Writes: lots to hash. Parallelism beats the overhead → PBT wins 1.43×. This is real and locality-independent — the parallel zone commit is the structural win we can claim.

Mixed: a blend. The signal is too small for 30-block CIs to call clearly.

Decomposing the headline ratio

For the read benchmark, per-slot total time and per-cache-miss are both ~1.0 — PBT and UBT pay the same per cold disk read and the same per slot once you remove the per-block fixed cost. The throughput gap (0.84×) is therefore entirely per-block fixed overhead amortised over a small block, not a per-operation slowness. In a workload where reads were embedded in much larger blocks (with real writes alongside, or much higher gas budgets per tx), the same per-block fixed cost would dilute — mixed (with writes paying for hashing) lands at 1.05× for exactly this reason.

What this says about the concentrated campaign

The earlier concentrated campaign (PBT 1.75× reads / 2.83× writes) measured the sum of (a) PBT's disk-locality advantage, (b) its parallel-commit advantage, and (c) block-packing artefacts. This campaign separates them:

  • Disk-locality is workload-dependent: present when many slots of one contract are swept (real ERC20 reads against one popular token), absent when access scatters across contracts (many tokens, many holders touching each only briefly).
  • Parallel-zone commit is structural: PBT wins ~1.4× on writes regardless of access pattern.
  • 3-zone root overhead is structural in the opposite direction: small read-only blocks pay a fixed PBT tax.
Section 8

Verdict

PBT's edge is two distinct effects, not one number.

A clean ~1.4× write speedup from parallel-zone commit that we can claim as structural, plus a workload-dependent disk-locality bonus on reads that ranges from large (1.75× when access concentrates on one contract) to negative (0.84× when it scatters across many). For a network where one popular contract gets swept hard, PBT wins big across the board; for a network where activity spreads across many contracts touching little storage each, the read story flips.

Caveats worth keeping in mind

  • The targets are getters, not ERC20s. Synthetic 28-byte contracts doing raw SLOAD/SSTORE, not ERC20 balanceOf/approve — chosen for scale (2000 deploys in ~20 min via factorydeploytx vs ~9 days via spamoor's erc20_bloater). For the across-contract clustering question this is a valid probe (1 slot/contract, what matters is H(addr)), but it's not a faithful ERC20-semantics workload.
  • Sample size is moderate. 30 cold 1-tx blocks per cell. Adequate for the headline effects (n=29 each side easily resolves 0.84× or 1.43×), tighter than ideal for the mixed result.
  • Reads happen in tiny blocks. Because we use one tx per block of ~16M gas and reads are cheap per slot, read blocks total ~90 ms. The per-block fixed overhead therefore looms larger than it would in a real production block packed with mixed work.

Open questions

  • Does PBT's 3-zone root overhead dilute to negligible in realistic mixed-work blocks (large gas, many writes), making the read penalty disappear in production?
  • Can UBT recover most of the concentrated read win with a locality-preserving key derivation without full zone partitioning?
  • Does the parallel commit advantage hold or grow further with more zones / deeper trees?
